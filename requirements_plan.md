# 运动记录App需求计划

## 项目概述
创建一个运动记录应用，用户可以通过日历界面记录每日运动情况和时长，并接收每日运动提醒。

## 需求收集和分析步骤

### 第一阶段：基础功能澄清
- [x] 1. 确认日历交互方式和用户体验
  - [Question] 用户点击日历某一天后，弹出的选择界面应该包含哪些选项？是简单的"是/否"选择，还是需要更详细的运动类型选择？
  - [Answer] 是/否

- [x] 2. 明确运动时长记录方式
  - [Question] 运动时长的选择方式是什么？是预设的时间段（如15分钟、30分钟、1小时）还是允许用户自定义输入具体分钟数？
  - [Answer] 预设的时间段

- [x] 3. 确认数据存储和历史记录查看
  - [Question] 用户如何查看历史运动记录？日历上是否需要显示已记录运动的日期（如用不同颜色标记）？
  - [Answer] 已记录信息直接显示在日历上，用红色标记

### 第二阶段：通知功能详细设计
- [x] 4. 通知功能具体行为
  - [Question] 每天11:30的通知是否需要用户交互？用户可以直接在通知中回复是否运动，还是需要打开app？
  - [Answer] 打开app

- [x] 5. 通知设置和个性化
  - [Question] 用户是否可以自定义通知时间？是否可以关闭通知功能？
  - [Answer] 不能自定义

### 第三阶段：用户界面和体验
- [x] 6. 日历显示范围和导航
  - [Question] 日历应该显示月视图还是周视图？用户如何导航到不同的月份？
  - [Answer] 月视图，通过翻页到下一个月份

- [x] 7. 运动记录的编辑和删除
  - [Question] 用户是否可以修改或删除已记录的运动数据？如果可以，操作方式是什么？
  - [Answer] 不可以删除修改

### 第四阶段：非功能性需求
- [x] 8. 平台和技术要求
  - [Question] 这个app是针对哪个平台？iOS、Android还是Web应用？有特定的技术栈要求吗？
  - [Answer] Android

- [x] 9. 性能和数据要求
  - [Question] 预期用户数量和数据量是多少？需要考虑多长时间的历史数据存储？
  - [Answer] 用户量10，数据量1000条，不需要长时间存储直接存在在app手机端

- [x] 10. 用户体验和可访问性
  - [Question] 是否需要考虑无障碍访问功能？应用的目标用户群体是什么？
  - [Answer] 不需要

### 第五阶段：补充澄清问题
- [x] 11. 预设时间段具体选项
  - [Question] Updated: 预设的运动时长选项具体包括哪些？比如15分钟、30分钟、45分钟、1小时、1.5小时、2小时等？
  - [Answer] 20分钟、30分钟、40分钟

- [x] 12. 日历显示的运动信息详细程度
  - [Question] Updated: 日历上用红色标记显示已运动的日期时，是否还需要显示具体的运动时长？比如显示"已运动 30分钟"还是仅显示"已运动"？
  - [Answer] 已运动 30分钟

- [x] 13. 通知内容和行为
  - [Question] Updated: 11:30的通知具体显示什么内容？用户点击通知后是直接进入当天的运动记录界面还是app主界面？
  - [Answer] 显示：哥哥，到时间运动了，今天是否运动？ 直接进入当天的运动记录界面

- [x] 14. 应用启动和默认界面
  - [Question] Updated: 用户打开app时默认显示哪个月份的日历？是当前月份还是有其他逻辑？
  - [Answer] 当前月份

- [x] 15. 数据持久化细节
  - [Question] Updated: 数据存储在手机端是指使用本地数据库（如SQLite）还是简单的文件存储？需要考虑app卸载重装后数据恢复吗？
  - [Answer] 简单的文件存储，需要考虑app卸载重装后数据恢复
